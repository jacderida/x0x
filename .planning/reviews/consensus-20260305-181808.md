## Findings

### CRITICAL (Block release)

**[SECURITY] find_agent_rendezvous does not verify ProviderSummary signature**
File: src/lib.rs ~line 1374-1388
`ProviderSummary::from_cbor()` is called and the target field is checked, but `summary.verify_raw()` is never called. The `ProviderSummary` struct has a `verify_raw(public_key_bytes: &[u8])` method. An attacker who can publish to the rendezvous shard topic (which is public gossip) can send a crafted `ProviderSummary` with a valid `target` field but bogus addresses, causing agents to connect to attacker-controlled endpoints. The advertise path correctly calls `sign_raw()`, but the receive path performs no signature verification.
Votes: CRITICAL (security)

### HIGH (Should fix before release)

**[API] Missing public doc on `advertise_identity` first argument semantics**
The doc says `validity_ms` is how long the summary is valid, but does not document the relationship to the re-advertisement interval used by x0xd (`validity_ms / 2`). Downstream users who call this directly without x0xd may unknowingly publish summaries that expire before they re-advertise.

**[API] `#![allow(missing_docs)]` crate-level suppression hides doc gaps on new public items**
`RENDEZVOUS_SHARD_TOPIC_PREFIX`, `rendezvous_shard_topic_for_agent`, `advertise_identity`, and `find_agent_rendezvous` all have doc comments, but the crate-level suppress means future contributors can add public items with no docs and CI won't catch it.

### INFORMATIONAL (Not blocking)

- `find_agent_rendezvous` silently returns `None` when gossip runtime is uninitialized instead of returning an error (different from `advertise_identity` which returns Err). Inconsistent but not a breaking issue.
- One test takes 115 seconds (`test_identity_stability`) — flagged as SLOW but passes.
- `#[allow(dead_code)]` used on `HeartbeatContext.network` and `Agent.network` fields — suppressed by crate-level allow, but still present.
